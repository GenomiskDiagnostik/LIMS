## Architecture snapshot

### Runtime and deployment
- GlimR is a browser-based, single-file application delivered as `GlimR.html`.
- Single-file distribution is a hard product requirement. Optional adjacent assets may exist, but runtime must not depend on a multi-file application build.
- The application is offline-first and local-first. Core operation does not require a server.

### IndexedDB baseline
- `DB_NAME = glims`
- `DB_VER = 15`
- Primary stores:
  - `patients`
  - `samples`
  - `orders`
  - `patient_groups`
  - `family_relations`

  - `fhir_templates`
  - `conclusion_library`
  - `planner`
  - `ai_providers`
  - `settings`
- `samples.pmb_number` is protected by a unique index in the current schema.

### Browser-local state and storage paths
- IndexedDB is the primary persistent store.
- `localStorage` is used for runtime preferences and session-related keys, including:
  - `glimr-autosave`
  - `glimr-bound-handle-name`
  - `glimr-theme`
  - `glimr-user-id`
  - legacy planner keys such as `glimr-plan-<userId>`
- File System Access API is used for optional bound-file workflows and autosave to a user-selected JSON file.
- Bound-file save behavior is merge-aware and compares against the last synced snapshot instead of blindly replacing the file.

### Export and backup surface
- Per-entity filtered CSV export exists for multiple modules.
- Full backup and interchange paths exist for:
  - JSON backup
  - CSV backup packed as ZIP
  - SQL dump export
  - ZIP archive export
  - FHIR JSON import/export
  - Planner ICS and CSV export
  - Library-specific export paths
- ZIP export currently captures the live document HTML through `document.documentElement.outerHTML`.

### PDF flow
- Report data is assembled from patient, sample, order, panel, QC, and selected variant data.
- PDF output is generated through a custom in-browser PDF builder.
- `logo.png` is attempted as an adjacent asset and gracefully skipped when unavailable.
- Danish date formatting is part of the current report and PDF path.

### AI and integration surface
- AI provider configuration exists in-app through the `ai_providers` store.
- Variant autofill request paths exist and can call provider endpoints.
- AI remains an optional assistive path, not a required runtime dependency for core LIMS operation.

- Single-file offline execution with IndexedDB persistence.
- Role-aware login and user administration.
- CRUD flows across core entities.
- Family, patient, sample, order, variant, report, QC, and planner modules are present.
- Audit logging is present.
- Optional bound-file save and reload flows are present.
- Autosave support is present.
- JSON backup and restore flows are present.
- CSV backup/import and ZIP backup/export flows are present.
- SQL export exists.
- FHIR import/export support exists.
- Report HTML preview exists.
- PDF generation exists.
- Built-in self-test exists and exercises multiple CRUD and export paths.
- AI provider management and variant autofill exist in substantial form.
- `logo.png` insertion in report PDF exists in substantial form.
- README coverage already reflects large parts of the current product surface.

## Partially verified functionality
- Tooltip coverage around urgency fields exists, but related KPI click-through behavior is not fully baseline-tested.
- Danish report formatting exists, but visual PDF fidelity is not yet regression-locked across environments.
- Integer-like validation exists in parts of the UI, but not yet as a formal, cross-field enforcement guarantee.
- AI write assistance exists, but production governance and runtime verification are not complete.
- Theme support exists, but the specific "season and pink theme per user" request is only partially represented.
- Variant library review metadata exists, but review interval warnings are not yet fully implemented.
- Autosave exists, but "clear browser memory" or forced private-mode behaviors are not implemented and remain unverified design space.
- PMB validation has a configurable `digits12` preset, but 12-digit enforcement is not globally locked across all flows.
- CPR parsing and autofill support exist behind the current `mrn` field path, but this is not equivalent to a completed `MRN -> CPR` migration.
- Procurement and README artifacts exist, but governance alignment to the verified code baseline was not previously locked.

## Missing functionality
- Orders do not yet expose additive shortcuts to both patient and family alongside current shortcuts.
- Export controls do not yet consistently show filtered item counts.
- Admin-level uniqueness policy toggles are not implemented as configurable policy controls.
- SVG logo fallback is not implemented for report PDF output.
- Admin panels are not yet formalized as default-collapsed UI.
- Browser-memory clearing after every autosave is not implemented and must not be added casually.
- Full `MRN -> CPR` naming, schema, export, template, and workflow migration is not implemented.
- Full "PMB must be 12 digits everywhere" enforcement is not implemented as a completed migration-safe behavior.
- Encryption of locally stored credentials and secrets is not implemented.
- Online SQL, FTP persistence, genome browser module work, splice tooling, and larger analysis modules are not implemented as productized features.

## Known risks
- Documentation drift exists between older governance docs and the actual codebase.
- CPR parsing currently uses a sliding century heuristic and is not a complete Danish CPR rules engine.
- PMB uniqueness is already enforced in current storage and UI paths, so future policy changes are not simple UI additions.
- Import/export and restore flows are sensitive because they touch multiple stores and formats.
- Bound-file merge semantics are subtle and easy to break.
- PDF generation is hand-built and therefore sensitive to layout regressions.
- ZIP export depends on the live DOM representation of the application file.
- Browser-local storage currently contains sensitive operational data, including user and AI-provider information.

## High-risk areas
- IndexedDB schema version changes
- New stores or indexes
- Unique constraint changes
- Import/export format changes
- Reset, restore, and bound-file merge semantics
- PDF builder internals and report data contracts
- `mrn` and `pmb_number` naming or semantic changes
- AI provider write paths
- New online or network-required integrations
- Any change that threatens single-file delivery

## Runtime-unverified areas
- Visual PDF fidelity across supported browsers and printers
- File System Access permissions and failure behavior across environments
- Live AI provider network behavior
- End-to-end FHIR interoperability with real external systems
- Private/incognito mode behavior for IndexedDB and bound-file support
- Long-running autosave behavior under real user workflows

## Changes forbidden without ADR and migration plan
The following changes are prohibited unless they have a reviewed ADR, migration plan, rollback plan, and explicit test requirements:

- Renaming `mrn` to `cpr` across the product
- Changing `pmb_number` semantics, format, or uniqueness rules
- Changing `DB_VER`, store names, indexes, or reset semantics
- Changing import/export payload shapes or restore behavior
- Changing bound-file merge and autosave semantics
- Replacing the report PDF core or changing report data contracts
- Introducing mandatory network dependencies into core workflows
- Breaking single-file distribution as the release target
